# Контекст продукта: автоматизация обогащения данных по ИНН и телефону (Tg_mtproto)

## 1. Что это за продукт

Набор сценариев на Python, которые **автоматизируют поиск контактов и краткой справки** по контрагенту:

- по **ИНН** — найти телефон (и связанные данные) через ответы **внешнего Telegram-бота** и при необходимости через **веб-отчёт** в браузере;
- по найденному **телефону** — запросить у того же внешнего бота **краткую сводку** (ФИО, возраст, Telegram, email, соцсети и т.п.);
- вход — **таблица** (CSV или Excel) или **одиночные запуски** из консоли;
- выход — **таблицы с результатами** (в т.ч. объединённый отчёт пайплайна).

Продукт **не** является самостоятельным «личным кабинетом» в браузере: это **инструмент оператора** (консоль + Telegram + при сценарии ИП — окно/фон браузера).

## 2. Технология (кратко)

Python-скрипты: **Telethon** (пользовательская сессия Telegram), **Telegram Bot API** (отдельный бот для приёма/выдачи файлов), **Playwright** (открытие веб-страницы отчёта для ИП), **openpyxl** / CSV для таблиц, **python-dotenv** для настроек.

## 3. Роли и доступ (если видно из кода)

- **Оператор с Telegram-аккаунтом** — от имени пользователя общается с **внешним ботом** (через сохранённую сессию после QR-входа). Без этой авторизации сценарии с ботом не работают.
- **Пользователь file-bot режима** — пишет в **конкретный Telegram-чат**, разрешённый настройкой `ID_TG_CHAT`; прикрепляет файл. Обрабатываются **только документы** из этого чата; чужие чаты игнорируются.
- **Разработчик / администратор** — настраивает `.env`, запускает `qr_login.py`, при необходимости узнаёт id чата через утилиту.

Отдельных «ролей в ЛК» (админ/менеджер в вебе) в коде **нет**.

## 4. Разделы и экраны (карта)

| Где пользователь видит интерфейс | Что это в продукте |
|----------------------------------|-------------------|
| **Чат с внешним ботом** (Telegram, клиент пользователя) | Источник данных: команды вроде `/inn`, карточки компаний и физлиц, inline-кнопки, ответ со ссылкой на отчёт, текст сводки по номеру. Скрипты имитируют действия пользователя в этом чате. |
| **Терминал / консоль** | Запуск `run_pipeline.py`, отдельных сценариев (`get_*`, `qr_login.py`), просмотр логов. |
| **Чат с собственным ботом** (file-bot) | Отправка входного `csv`/`xlsx`/`xlsm`; сообщения о статусе обработки; получение итогового **xlsx**. |
| **Браузер (Playwright)** | Только для цепочки **ИП**: открытие **веб-страницы отчёта** по ссылке из Telegram. Режим окна задаётся настройкой (headless / с окном). |

Отдельных URL-маршрутов SPA **нет**.

## 5. Сущности и связи (бизнес-языком)

- **Контрагент / запись списка** — строка входной таблицы: как минимум **наименование** (первый столбец) и **ИНН** (второй столбец), либо особый случай (см. сценарии).
- **Тип контрагента** — по тексту наименования определяется **компания** или **ИП** (по шаблонам вроде «ИП», «индивидуальный предприниматель»); иначе считается **компания**.
- **ИНН** — 10 или 12 цифр, извлекается из ячейки.
- **Телефон** — найденный контакт или введённый для прямой сводки (формат в коде: российский **11 цифр, начинается с 7** в первом столбце).
- **Краткая сводка по телефону** — ответ внешнего бота с полями профиля (ФИО, дата рождения, возраст, Telegram, email, ИНН, телефонные книги, ВК, Instagram, OK — насколько бот их отдаёт).
- **Задача обработки (job)** — при file-bot каждый запуск складывает файлы во вложенную папку в рабочем каталоге заданий.

## 6. Типовые сценарии (по коду)

1. **Первичная привязка Telegram-пользователя** — запуск `qr_login.py`, сканирование QR, при 2FA ввод пароля; создаётся файл сессии для Telethon.
2. **Пакет из файла (основной пайплайн)** — `run_pipeline.py` читает таблицу; для каждой строки: если во втором столбце есть ИНН — ветка **компания** или **ИП**; если в первом столбце распознан прямой телефон `7…` — только **сводка по номеру** без поиска по ИНН.
3. **Компания** — отправка боту `/inn` и ИНН, обход карточек и **inline-кнопок** до данных **физлица**, извлечение телефона, email, ИНН физлица; затем запрос сводки по найденному телефону.
4. **ИП** — отправка `/inn`, получение **ссылки на полный веб-отчёт**, открытие страницы в Playwright, разбор **первой личности** и контактов; при сбоях возможно сохранение отладочных артефактов страницы.
5. **Только сводка по телефону** — отдельный сценарий или строка таблицы с телефоном в первом столбце: отправка номера боту, разбор ответа, сохранение в свои файлы результатов [в режиме одиночных скриптов — см. readme; в пайплайне — колонки итоговой таблицы].
6. **File-bot** — пользователь в нужном чате отправляет файл; бот скачивает его, гонит тот же пайплайн по строкам, шлёт прогресс в чат, возвращает **готовый xlsx**.

Между шагами и строками предусмотрены **паузы** (настраиваемые задержки), чтобы не перегружать бота.

## 7. Интеграции и внешние системы (если видно)

- **Внешний Telegram-бот** (username задаётся в `BOT`) — основной источник карточек и сводок. Логика ответов бота **не** в этом репозитории; проект только автоматизирует диалог.
- **Telegram** как транспорт для собственного бота (`TG_BOT_TOKEN`) — приём файлов и отправка результата.
- **Произвольный веб-сайт отчёта** — открывается по ссылке из бота (сценарий ИП); содержимое парсится локально.

## 8. Ограничения и неочевидное (помечать как гипотезу или «уточнить»)

- Входная таблица пайплайна: **первый столбец — наименование (или телефон `7…`)**, **второй — ИНН**; первая строка может быть **шапкой** (эвристика по словам «название», «ИНН» и т.д.).
- Поддерживаемые расширения file-bot: **`.csv`, `.xlsx`, `.xlsm`**.
- Итог пайплайна по умолчанию: **`pipeline_results.csv`** и **`pipeline_results.xlsx`** (имена можно переопределить переменными окружения).
- Статусы итога включают по смыслу: успех с сводкой, телефон без сводки, ошибка поиска телефона, ошибка ввода (нет ИНН), и т.д. — детали в колонках `pipeline_status`, сообщениях.
- В `.env.example` указан **`DB_PATH`**, но **в Python-файлах репозитория не используется** — [неясно по коду — уточнить у владельца продукта], не опираться на наличие БД при описании ТЗ.
- Юридические основания обработки данных и лимиты внешнего бота **из кода не следуют** — [неясно по коду — уточнить у владельца продукта].

## 9. Глоссарий

| Термин | Пояснение |
|--------|-----------|
| Пайплайн | Цепочка: ИНН → телефон (и данные лица) → сводка по телефону; запись в итоговую таблицу. |
| Внешний бот | Telegram-бот, который отдаёт справочные карточки и сводки; с ним работает Telethon от имени пользователя. |
| File-bot | Собственный бот: принимает таблицу в заданном чате, отдаёт результат xlsx. |
| Сессия (Telethon) | Файл после `qr_login.py`, привязанный к аккаунту Telegram пользователя. |
| Сценарий компании | Обход карточек и кнопок в Telegram до физлица. |
| Сценарий ИП | Переход по ссылке на **веб-отчёт** и разбор страницы. |
| Сводка по телефону | Расширенный профиль по номеру из ответа внешнего бота. |
| `ID_TG_CHAT` | Числовой id чата, из которого file-bot принимает файлы. |
| `tg_bot_jobs/` | Рабочие папки обработок file-bot (по умолчанию). |
| `report_debug/` | Сохранение материалов для разбора сбоев парсинга веб-отчёта (по умолчанию). |

## Что агенту ТЗ не стоит выдумывать

- Назначение и правила **внешнего бота** (что именно он вернёт на `/inn` и на номер) — это вне репозитория; при сомнениях: **[неясно по коду — уточнить у владельца продукта]**.
- Наличие и использование **БД** (`DB_PATH`) — в коде не подтверждено.
- **Юридическая сторона** (согласия, КПД, политика хранения) — не выводятся из кода.
- Точные **лимиты Telegram**, стабильность вёрстки **чужого сайта** отчёта — гипотезы; менять ожидания только по ТЗ заказчика.
- Любые **новые экраны веб-ЛК** — в этом проекте отсутствуют; не описывать как часть текущего продукта без отдельного репозитория.
